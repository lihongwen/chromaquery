# 文件上传进度同步修复测试指南

## 修复内容总结

### 前端改进
1. **超时设置优化**
   - 普通API请求：30秒 → 60秒
   - 文件上传请求：30秒 → 300秒（5分钟）
   - 为文件上传创建专门的axios实例

2. **进度显示改进**
   - 移除模拟进度，改为基于文件大小和类型的智能估算
   - 添加剩余时间显示
   - 根据文件类型提供不同的处理提示

3. **错误处理优化**
   - 区分超时错误和真正的处理失败
   - 为超时情况提供更友好的提示
   - 建议用户稍后刷新查看结果

### 后端改进
1. **详细日志记录**
   - 文件读取阶段的时间记录
   - 文件解析阶段的时间记录
   - RAG分块阶段的时间记录
   - 向量化和存储阶段的时间记录

2. **处理流程优化**
   - 在关键阶段添加进度日志
   - 记录每个阶段的耗时

## 测试步骤

### 1. 小文件测试（< 1MB）
- 上传一个小的文本文件
- 验证进度显示是否合理（约10-20秒完成）
- 检查是否有超时错误

### 2. 中等文件测试（1-10MB）
- 上传一个中等大小的PDF或Word文档
- 验证进度估算是否准确
- 检查各阶段切换是否正常

### 3. 大文件测试（50-150MB）
- 上传一个接近限制的大文件
- 验证不会出现30秒超时错误
- 检查进度显示是否与实际处理同步

### 4. 不同文件类型测试
- PDF文件：验证解析时间估算
- Excel文件：验证表格处理提示
- Word文档：验证格式解析提示

## 预期结果

### 成功指标
1. 大文件上传不再出现30秒超时错误
2. 进度显示与实际处理阶段基本同步
3. 用户能看到准确的剩余时间估算
4. 超时时提供友好的提示而非错误报告

### 日志检查
后端日志应该显示：
```

文件读取完成，大小: 45.67MB
开始解析文件: large_file.pdf (45.67MB)

开始RAG分块处理，文本长度: 123456 字符，方法: recursive

向量化和存储完成，耗时: 23.45秒
文件处理完成: large_file.pdf, 总耗时: 45.67秒, 创建文档块: 156
```

## 故障排除

### 如果仍然出现超时
1. 检查文件大小是否超过150MB限制
2. 检查网络连接是否稳定
3. 查看后端日志确认处理进度
4. 考虑分割大文件后重新上传

### 如果进度显示不准确
1. 检查文件类型是否在支持列表中
2. 验证估算算法是否适合当前文件
3. 根据实际处理时间调整估算参数
